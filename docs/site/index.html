01 — The problem

You can't see the cliff
until you're falling.

"Past ~80% context utilisation, the model attends equally to stale retries and abandoned threads as it does to your original goals. Quality degrades. Silently."

Every Claude session has a context efficiency curve. In the early portion, each input produces high-quality, well-grounded output. Past the garbage threshold — typically 70–80% of the context window — signal-to-noise ratio collapses. This threshold is currently invisible to users. Claude OS makes it visible, measurable, and actionable.

03 — Get started

Three ways in.

Choose the path that fits your workflow. Most setups combine all three — backfill once to seed history, wire the hook for live data, reach for the wrapper when you need compaction.

Path 01 — Backfill
One-time historical import

Run once to bootstrap the database with every Claude session you've ever had. Instantly see full history in the Activity Monitor — no code changes required.

bun run ingest # walks ~/.claude/projects/ # idempotent — safe to re-run
  • ✓ Full session history
  • ✓ ctx depth + GC states for past work
  • — No live updates going forward
Path 02 — Zero-config live
Ongoing hook instrumentation

Wire one Stop hook entry in ~/.claude/settings.json. Every Claude Code turn writes to SQLite automatically — no API key, no interruptions, no code changes.

bun run setup # registers hook-stop.ts in # ~/.claude/settings.json
  • ✓ Per-turn live data as you work
  • ✓ Zero friction — fires on every stop
  • — No policy-driven compaction
Path 03 — SDK wrapper
Programmatic instrumentation + compaction

Wrap your Anthropic client directly. Same metrics as the hook, plus policy-driven compaction — automatic memory distillation on context thresholds or semantic triggers.

import { createInstrumentedClient } from "@claude-os/core"; const client = createInstrumentedClient( process.env.ANTHROPIC_API_KEY, { cwd: process.cwd() } );
  • ✓ Full live pipeline + getHealth()
  • ✓ Policy-driven compaction on trigger
  • ✓ Custom memory schemas per project
04 — GC taxonomy

Four states. One state machine.

Transitions are one-way within a session. A forked session starts Clean. Hard GC triggers a one-click Compact & Fork — auto-compaction to a structured markdown summary, seeded into a new session as context.

Clean ctx < 60%

Full signal fidelity. Context accumulation is additive. No action needed.

→ continue normally
Soft GC ctx 60–80%

Signal degradation beginning. Stale context accumulating. Consider compaction.

→ summarise thread, compact
Hard GC ctx > 80%

Garbage threshold breached. Quality impact measurable. Fork recommended.

→ compact & fork session
Aged closed, unresolved

Session closed without outcome resolution. Archived to persistent memory.

→ export to CLAUDE.md
05 — Roadmap

Six phases.

From a CLI proof-of-concept to an open-source, Homebrew-installable macOS utility. With an MCP server surface so Claude can read its own context health.

Phase 0
✓ shipped
Proof of Concept
  • Anthropic Messages API wrapper — per-turn token capture
  • Session-level context utilisation % in real time
  • Quality proxy logging: length, latency, self-correction rate
  • SQLite store + efficiency curve notebook
Phase 1
✓ shipped
Menu Bar + Live Feed
  • Electron tray icon — real-time context depth, colour-coded GC state
  • Stop hook wired into Claude Code — per-turn SQLite writes
  • Hono API server on :7842, polling every 5s
Phase 2
✓ shipped
Activity Monitor Window
  • Full panel: session list, live efficiency curve chart
  • Electron + React renderer, macOS vibrancy material
  • Turn-by-turn breakdown, GC event log
Phase 3
✓ shipped
Compaction Engine
  • Policy-driven triggers: context threshold, turn cadence, semantic classifiers
  • Memory distillation — per-project memory files written on trigger
  • Unified turn pipeline: single source of truth for live + backfill metrics
Phase 4
▸ current
DX + Policy UI
  • Activity Monitor redesign — grouped by Project → Session, expandable context depth per session
  • Policy editor UI — configure and mutate compaction policies per project without touching JSON
  • Outcomes layer: cost-per-outcome reporting, stalled session detection
  • User-configurable outcome definitions per session type
Phase 5
Public Release + Research
  • MCP server: Claude can query its own context health in real time
  • Empirical write-up: efficiency curves across session types and models
  • Homebrew tap: brew install claude-os
